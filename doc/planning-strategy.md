# Planning 策略文档

## 概述

Planning 模块是活动规划 Agent 的核心，负责根据用户需求、偏好和可用数据生成完整的出行方案。本文档详细描述规划策略的设计理念、执行流程和核心算法。

---

## 一、规划策略架构

### 1.1 整体流程

```
用户输入 → 意图分析 → 偏好提取 → 数据查询 → 规划生成 → 交通后处理 → 方案输出
```

### 1.2 核心组件

| 组件 | 职责 | 关键文件 |
|------|------|----------|
| **意图分析** | 识别用户意图类型，匹配 Skill | `agent/nodes/intent.py` |
| **偏好提取** | 提取用户偏好并持久化 | `agent/nodes/preferences.py` |
| **数据查询** | 根据约束条件查询可用数据 | `agent/nodes/query.py` |
| **规划生成** | LLM 生成结构化方案 | `agent/nodes/planner.py` |
| **交通计算** | 计算活动间交通方式和时间 | `core/travel.py` |

---

## 二、规划生成策略

### 2.1 核心算法

#### 2.1.1 基于 LLM 的规划生成

规划生成采用 **提示词工程 + LLM 推理** 的方式：

```python
# 核心流程
1. 构建 System Prompt（规划约束、输出格式）
2. 构建 User Prompt（用户需求、偏好、可用数据）
3. 调用 LLM 生成 JSON 格式方案
4. 解析并验证 JSON
5. 后处理（插入交通、补全信息）
```

#### 2.1.2 输出格式约束

LLM 必须输出严格的 JSON 格式：

```json
[
  {
    "order": 1,
    "time_slot": {"start": "09:00", "end": "11:00"},
    "activity_type": "attraction",
    "name": "活动名称",
    "item_id": "数据中的ID",
    "location": {"lat": 39.9, "lng": 116.4, "address": "地址"},
    "duration_hours": 2,
    "details": {"rating": 4.5, "price": 50, "tags": ["标签"]},
    "pre_book": {"need": false, "type": "ticket", "item": "描述"}
  }
]
```

### 2.2 时间规划策略

#### 2.2.1 时间约束规则

| 约束类型 | 规则 |
|----------|------|
| **开始时间** | 默认上午 9:00，可根据用户偏好调整 |
| **用餐时间** | 午餐 11:30-13:00，晚餐 17:30-19:30 |
| **活动间隔** | 前后活动时间不重叠 |
| **亲子场景** | 每 2 小时穿插休息/饮品环节 |

#### 2.2.2 行程密度控制

- **半天行程**：2-3 个活动
- **全天行程**：4-6 个活动
- **活动时长**：参考数据中的 `duration_hours` 或 `visit_duration_hours`

### 2.3 交通规划策略

#### 2.3.1 交通方式选择算法

基于 Haversine 球面距离公式计算两点间距离，然后根据距离选择最优交通方式：

| 距离 | 推荐交通 | 速度 | 固定开销 |
|------|----------|------|----------|
| < 800m | 步行 | 5 km/h | 无 |
| 800m-3km | 打车 | 35 km/h | +5min（等车） |
| 3km-15km | 公共交通/打车 | 25/35 km/h | +8min（等车/走路） |
| > 15km | 自驾/公共交通 | 30/25 km/h | +8min（取车/停车） |

#### 2.3.2 交通活动注入

交通作为独立的 `activity_type: "transport"` 节点插入到活动之间：

```python
# 输入: [活动A, 活动B, 活动C]
# 输出: [活动A, 交通A→B, 活动B, 交通B→C, 活动C]
```

交通活动结构：

```json
{
  "activity_type": "transport",
  "name": "打车前往海底捞火锅",
  "from_location": {"lat": 39.942, "lng": 116.343, "name": "北京动物园"},
  "to_location": {"lat": 39.943, "lng": 116.355, "name": "海底捞火锅"},
  "duration_minutes": 22,
  "distance_m": 10043,
  "mode": "taxi",
  "mode_label": "打车",
  "mode_icon": "🚕"
}
```

### 2.4 多样性约束策略

#### 2.4.1 类型多样性

- 同一类型餐厅最多出现 1 次
- 餐饮和景点/活动交替安排
- 优先选择评分高、适合用户场景的地点

#### 2.4.2 预订需求识别

| 条件 | 预订需求 |
|------|----------|
| 景点 `need_ticket=true` | `pre_book.need=true` |
| 餐厅评分 >= 4.5 | `pre_book.need=true` |
| 普通咖啡馆 | `pre_book.need=false` |

## 三、重规划策略

### 3.1 三种重规划模式

#### 3.1.1 替换单活动 (replace_activity)

**触发条件**：用户说"换掉XX"、"第二个不适合"

**执行流程**：
1. 定位目标活动（关键词匹配 → LLM 兜底）
2. 计算约束条件（时间窗口、位置范围）
3. 查询替代数据
4. 选择最优替代（评分 + 距离综合评分）
5. 更新方案，重算受影响的交通

**约束求解**：
- 替换后的活动时长 ≤ 原活动时长
- 位置必须在前后活动可达范围内
- 不影响前后活动的时间安排

#### 3.1.2 部分重规划 (partial_replan)

**触发条件**：用户说"从XX之后重新规划"

**执行流程**：
1. 确定分割点（按名称匹配或默认保留前半部分）
2. 保留分割点之前的活动
3. 从分割点结束时间和位置开始重新规划
4. 合并保留部分和新规划部分

#### 3.1.3 全量重规划 (full_replan)

**触发条件**：用户说"全部重新规划"

**执行流程**：
1. 清空现有方案
2. 将意图类型改为 `planning`
3. 触发完整的规划流程

### 3.2 目标活动定位策略

#### 3.2.1 多级匹配机制

```
直接索引 → 名称匹配 → 类型搜索 → LLM 兜底
```

#### 3.2.2 LLM 兜底逻辑

当关键词无法精确定位时，使用 LLM 分析用户意图：

```python
prompt = f"""
用户说："{user_input}"
当前规划方案中的活动列表：
{plan_summary}
请判断用户想替换的是哪一个活动。只返回活动序号。
"""
```

### 3.3 替代方案选择算法

#### 3.3.1 评分公式

```
score = rating * 10 - distance / 1000
```

#### 3.3.2 约束过滤

在选择替代活动时，需满足以下约束：
- 时间约束：往返交通时间 + 活动时长 ≤ 可用时间窗口
- 位置约束：在前后活动位置的可达范围内

## 四、Skill 场景化策略

### 4.1 Skill 匹配机制

#### 4.1.1 匹配流程

```
用户输入 → 关键词提取 → Skill 匹配 → 注入 Prompt
```

#### 4.1.2 Skill 优先级

| Skill ID | 场景 | 触发关键词 |
|----------|------|------------|
| `Parent-ChildTravelPlanning` | 亲子出行 | 孩子、亲子、带娃 |
| `FriendsOuting` | 朋友聚会 | 朋友、聚会、团建 |
| `PersonalTravelPlanning` | 个人出行 | 默认、无特定关键词 |

### 4.2 Skill 上下文注入

匹配到的 Skill 会注入到 Planner 的 System Prompt 中：

```python
if skill_context:
    system_prompt += "\n\n## 场景专项指导\n" + _build_scenario_description(skill_context)
```

注入内容包括：
- 业务流程
- 注意事项
- 输出风格参考

## 五、偏好持久化与复用

### 5.1 偏好提取策略

从用户消息中提取以下类型信息：

| 类别 | 提取内容 |
|------|----------|
| 家庭信息 | 孩子年龄、家庭成员 |
| 饮食偏好 | 口味偏好、忌口、饮食限制 |
| 出行偏好 | 交通方式、活动类型 |
| 预算偏好 | 消费预算范围 |

### 5.2 持久化机制

```
用户输入 → extract_preferences → 合并已有偏好 → 写入 preferences.json
```

偏好数据用于：
- 每次规划时注入 Prompt
- 个性化推荐
- 场景匹配

## 六、性能优化策略

### 6.1 数据缓存

- 查询结果缓存到 `state.query_results`
- 同一会话内重复查询直接使用缓存

### 6.2 LLM 调用优化

- 流式输出：减少用户等待时间
- 多 LLM 轮询：提高可用性
- 自动重试：处理临时故障

### 6.3 约束剪枝

在查询替代方案时，提前过滤不符合约束的选项，减少计算量。

---

## 七、边界情况处理

### 7.1 数据为空

当某类别数据为空时：
- 跳过该类别活动
- 在方案中说明该类别暂无数据

### 7.2 时间冲突

检测到时间冲突时：
- 自动调整活动顺序
- 提示用户存在时间冲突

### 7.3 位置不可达

当活动间距离过远时：
- 建议更换活动
- 提示用户调整行程
